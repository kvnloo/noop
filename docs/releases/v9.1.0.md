# NOOP v9.1.0

A reliability-and-accuracy release on top of 9.0.2. The headline is **workout HR/calorie backfill** for manual entries, a **strap model fix** that restores skin temp for affected WHOOP 4.0 users, and a **clock retry** that prevents overnight data from mis-dating. Everything still runs on your own device, offline, no account. Update from the Releases page, AltStore, or `brew upgrade`.

## New

- **Diagnostic WHOOP GATT family detection (#688, #713).** NOOP now scans for and logs the additional WHOOP service families (Puffin, Monument, Symphony) seen in the decompiled app. Detected-but-unsupported straps get a clear log message instead of being silently missed. Existing WHOOP 4.0 and fd4b 5.0/MG connect behavior is unchanged.
- **WHOOP 5.0 SpO2 candidate instrumentation (#103, #709).** The v18 byte @82 is now decoded as an instrumentation-only SpO2 candidate (70-100 = in-band %, sentinels and diagnostics filtered). Not a shipped metric — evidence is split across devices — decoded raw for multi-device correlation.
- **Oura feature-status probe (#710).** A read-only diagnostic (the 0x20 READ verb, never an enable) asks the ring its SpO2 and real-steps feature status. Confirms from the ring itself that these are server-flag-gated off for an offline ring.
- **Configurable bundle ID and team for iOS builds (#701, #722).** A single gitignored xcconfig file (`Config/BundleIdSecrets.xcconfig`) now drives every target's bundle ID, App Group, and development team — no more hand-editing project.yml in six places after every `xcodegen generate`.

## Fixed

- **Seeded WHOOP device showed wrong model and broken skin temp (#716, #719).** The default "my-whoop" device had model "WHOOP" (no generation suffix), so `forRegistryModel` returned the wrong family. Skin temp used the wrong ADC scale and the Devices screen showed "Exact model unknown." Now stamped to the correct family on first BLE scan, on both platforms.
- **Manual workouts showed no HR or calories (#510, #708).** When the auto-workout detector found the same activity a user manually logged, it discarded its computed HR/calories instead of backfilling the manual entry. Fixed — missing fields (avgHR, maxHR, calories, strain) are now filled from the detector's own values without overriding anything the user typed or imported.
- **Journal import zeroed out every answer from a real WHOOP export (#631, #706).** The real WHOOP export column is "Answered yes" (normalized to `answered_yes`), not the "Answered yes/no" (`answered_yes_no`) NOOP's own exporter writes. Every imported journal entry silently read as false.
- **Sleep Schedule card drew phantom bars from night-tail fragments (#699, #703).** The Schedule card iterated raw stored sessions instead of the per-night bridged windows the rest of the Sleep tab uses. A briefly-interrupted or biphasic night's tail fragment drew as its own low bar, inflated the night count, and skewed the consistency score. Now bridges the same way the hero does.
- **Steps calibration slider was unusable (#698, #704).** The manual coefficient slider compressed the entire 0-100+ range into a few drag pixels. Replaced with a stepper (fixed 0.1 increments), matching the Profile card's age/weight pattern.
- **Backfill clock retry and Data Range fallback (#700, #721).** If the strap silently drops every GET_CLOCK response (observed on a reporter's WHOOP 4.0), NOOP now retries up to 3 times before backfill, then derives a rough correlation from the Data Range's newest-banked timestamp. Prevents all rows from landing on the current day under the identity fallback.
- **Oura IBI timestamps were wrong (#677).** Banked overnight beats were stamped at the drain-arrival wall-clock time instead of their real ring-time, so the sleep window had zero R-R data — no restingHr or avgHrv for the night.

## Diagnostics

- **Log why a day is skipped for sleep (#714, #720, #723).** When the analytics pass skips a day for having fewer than 200 HR samples, the strap log now says `sleep day=... SKIPPED hrSamples=N (need >=200)` — immediately answers "no data" vs "data but no sleep found."
- **Alarm HR diagnostic (#34, #707).** Live HR at alarm arm time is logged to help investigate the firmware alarm hypothesis (evening arms not firing).

## Infrastructure

- **Android Gradle supply chain hardened (#658, #683).** Dependency locking (`lockAllConfigurations`), strict SHA-256 verification for all downloaded artifacts, and a checked-in Gradle 8.7 wrapper with a pinned distribution checksum. The lockfile and verification metadata are committed; regeneration is documented.
- **Code quality refactors (#665, #686, #711, #712).** Today, Sleep, and Health screens' pure helper logic extracted into sibling files (TodayDayNav, TodayScoring, TodayProvenance, TodayMetricsLogic, SleepHeroLogic, SleepFormatting, SleepMetricDetailLogic, HealthLiveHrLogic, HealthVitalDetailLogic, HealthVitalsLogic) — no behavior change.
- **Oura protocol docs updated (#678).** IBI decode corrections, wear/charge signal documentation, observed-but-undecoded tags appendix, and the feature-status probe findings.
